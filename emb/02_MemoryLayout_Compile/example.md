# Memory Layout & Compile - 10 Case Studies

## Case 1: The Massive Local Array (Stack Overflow)
**Context**: Processing an image line on a thermal printer.
**Bad Practice**:
```cpp
void process_line() {
    uint8_t image_buffer[8192]; // 8KB allocated on stack!
    // ...
}
```
**Good Practice**:
```cpp
// Allocated in .bss (RAM), not on the stack.
static uint8_t image_buffer[8192]; 
void process_line() {
    // ...
}
```
**Explanation**: Small MCUs might only have 16KB of RAM total. Allocating 8KB on the stack will almost certainly cause a stack overflow and crash. Using `static` moves it to `.bss`.

## Case 2: Memory Fragmentation (The Evils of `malloc`)
**Context**: An IoT node parsing incoming JSON payloads of varying sizes.
**Bad Practice**: `char* payload = (char*)malloc(size); ... free(payload);`
**Good Practice**: Use a statically allocated maximum-size buffer. `char payload[MAX_PAYLOAD_SIZE];`
**Explanation**: ISO 26262 and MISRA forbid dynamic allocation (`malloc`/`free`, `new`/`delete`) because memory fragmentation over days/weeks of uptime will eventually cause `malloc` to fail, resulting in a system halt.

## Case 3: Saving RAM with `const` Data
**Context**: Storing HTML/CSS files on an ESP32 web server.
**Bad Practice**: `char index_html[] = "<html>...</html>";` (Takes up RAM).
**Good Practice**: `const char index_html[] = "<html>...</html>";` (Takes up Flash).
**Explanation**: Without `const`, the C startup code copies the string from Flash into RAM (`.data` section) at boot, wasting RAM.

## Case 4: Zero-Initialized vs Explicit-Initialized
**Context**: A global state machine variable.
**Bad Practice**: `uint8_t state = 0;`
**Good Practice**: `uint8_t state;` (If relying on C-standard) or explicitly init in `main()` for safety critical.
**Explanation**: `uint8_t state = 0;` puts the variable in the `.bss` section (zeroed at boot). `uint8_t state = 1;` puts it in `.data` (requires Flash storage for the init value '1' and boot-time copying).

## Case 5: Linker Placement for DMA Buffers
**Context**: A DMA controller needs to access a specific region of SRAM (e.g., SRAM2 on STM32).
**Practice**: You cannot do this purely in C. You must use compiler-specific attributes to tell the Linker Script to place the variable in a specific memory section.
```cpp
// GCC syntax
uint8_t dma_buffer[1024] __attribute__((section(".sram2_bss")));
```
**Explanation**: Hardware architecture often dictates where certain variables must live in physical memory.

## Case 6: Catching Code Size Issues (Map File)
**Context**: Firmware fails to flash because it's "too large".
**Action**: Inspect the `.map` file generated by the Linker.
**Explanation**: The `.map` file shows exactly how much Flash and RAM every function and variable consumes. It's the primary tool for optimizing memory layout.

## Case 7: Unused Function Elimination
**Context**: Including a huge math library but only using one function.
**Practice**: Compile with `-ffunction-sections` and `-fdata-sections`, and link with `--gc-sections`.
**Explanation**: These compiler/linker flags ensure that if a function or variable is never called, it is stripped from the final binary, saving Flash memory.

## Case 8: The Cost of `printf` and Floating Point
**Context**: Adding a debug print in a tiny 8-bit MCU.
**Bad Practice**: `printf("Temp: %f\n", 25.4);`
**Explanation**: Pulling in standard `printf` and floating-point support can instantly add 10-20KB of code to the `.text` segment (Flash). Use lightweight integer formatting functions instead (e.g., `snprintf` with integers: `Temp: %d.%d`).

## Case 9: Avoiding Recursion
**Context**: Traversing a menu tree on an LCD screen.
**Bad Practice**: A menu function that calls itself to go deeper.
**Good Practice**: Use a `while` loop and an iterative state machine.
**Explanation**: MISRA specifically bans recursion because it makes maximum stack usage impossible to calculate statically, leading to potential stack overflows in production.

## Case 10: Watchdog Timer (WDT) and Startup Time
**Context**: Large `.data` segment taking too long to copy at boot.
**Scenario**: If the `.data` segment is massive, the C startup code (`Reset_Handler`) might take too long to copy it from Flash to RAM. If the Hardware Watchdog is enabled early, it might reset the chip before `main()` even starts!
**Fix**: Keep `.data` small.
